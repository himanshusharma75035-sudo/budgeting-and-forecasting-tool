## 1. Research dossiers

### budgeting-methods
Corporate FP&A literature has two competing "4 budgeting methods" lists. The dominant strategic/zero-base lineage (Corporate Finance Institute, Peak Frameworks, Abacum, Milestone) names: (1) Incremental, (2) Activity-Based Budgeting (ABB), (3) Value Proposition / Priority-Based Budgeting, and (4) Zero-Based Budgeting (ZBB). The management-accounting/control lineage (AccountingTools, Lumen/PSU managerial accounting texts) instead emphasizes Static, Flexible, Rolling/Continuous, and Incremental — these classify budgets by how they FLEX with activity/time rather than by how line items are JUSTIFIED. The two lists are not contradictory: the CFI four answer "how do we DERIVE each line item?" (justification basis) while the control four answer "how does the budget RESPOND to volume/time?" (re-forecasting cadence). RECOMMENDATION: Implement the CFI canonical four (Incremental, ABB, Value-Proposition, ZBB) as the primary "budgeting methods" because they are mutually exclusive derivation engines that map cleanly to distinct, autonomously executable algorithms, and they are the set most explicitly cited as "the 4." Layer Rolling/Continuous and Flexible as orthogonal MODES (re-forecast cadence and volume-flexing) that can wrap ANY of the four — this resolves the conflict and is more powerful than treating them as mutually exclusive alternatives. All four primary methods can consume prior actuals and forecast drivers: Incremental and ABB consume them directly (baseline + uplift; driver volume x rate); ZBB and Value-Proposition use actuals/drivers as evidence for justification and ranking rather than as a mechanical baseline.

**Key findings**

- **Reconciling the two competing '4 methods' lists** — Two authoritative groupings exist. List A (strategic / justification-basis, used by Corporate Finance Institute, Peak Frameworks, Abacum, Milestone, The Finance Weekly): Incremental, Activity-Based, Value-Proposition, Zero-Based. List B (control / behavior-basis, used by AccountingTools and university managerial-accounting texts such as Lumen Learning and PSU/Unizin): Static, Flexible, Rolling/Continuous, Incremental (plus sometimes ZBB). The distinction: List A classifies methods by HOW a line item's amount is derived and justified (carry-forward+%, activity x rate, value ranking, or from-zero). List B classifies budgets by HOW the budget reacts to changes in activity volume (static=fixed; flexible=flexes with actual volume) or time (rolling=continuously re-extended). They answer different questions and are therefore composable rather than mutually exclusive. Incremental is the only method appearing on both lists, which is why it anchors any implementation.
  - `spec:` List A = {Incremental, ABB, Value-Proposition, ZBB} (derivation engines). List B = {Static, Flexible, Rolling, (Incremental)} (response modes). Recommended product model: 4 derivation engines x optional {Static|Flexible} volume mode x optional {Annual|Rolling} cadence.
- **Recommended canonical four to implement** — Implement Incremental, Activity-Based (ABB), Value-Proposition (Priority-Based), and Zero-Based (ZBB). Rationale: (1) these are the set most frequently and explicitly enumerated as 'the four types of budgets' by the leading FP&A reference (CFI) and corroborating sources; (2) each maps to a DISTINCT, deterministic algorithm a tool can run autonomously given structured inputs, whereas Static vs Flexible differ only by whether you re-multiply by actual volume (a toggle, not a separate engine), and Rolling differs only by scheduling cadence (a job, not a derivation); (3) treating Flexible and Rolling as orthogonal modes that wrap any of the four gives users e.g. 'Rolling ABB' or 'Flexible Incremental,' which is strictly more expressive than forcing a choice among 4 mutually-exclusive options.
  - `spec:` Primary enum: {INCREMENTAL, ACTIVITY_BASED, VALUE_PROPOSITION, ZERO_BASED}. Modifiers: volumeMode in {STATIC, FLEXIBLE}; cadence in {PERIODIC, ROLLING(horizon_periods)}.
- **Method 1 — Incremental Budgeting: definition, inputs, algorithm, formula, example** — DEFINITION (CFI): 'takes last year's actual figures and adds or subtracts a percentage to obtain the current year's budget'; LibreTexts: 'starts with the prior year's actual results and adjusts them upward or downward for inflation, growth, or strategic initiatives.' WHEN USED: stable operations with predictable cost drivers; default method when speed/simplicity matter; most common method in practice. INPUTS REQUIRED: prior-period actuals (or prior budget) per line item/cost center; an uplift assumption per line (or global) — inflation %, expected growth %, FX, headcount changes; optional one-off adjustments (new initiatives, discontinued items). ALGORITHM (autonomous): (1) Load prior-period actuals A_i for each line i. (2) For each line, resolve an adjustment factor g_i = base inflation + growth + strategic delta (or a flat global %). (3) Compute Budget_i = A_i x (1 + g_i) + one_off_adjustments_i. (4) Sum to totals; (5) optionally cap/floor; (6) emit for approval; (7) after actuals arrive, run variance = Actual - Budget to feed next cycle. CONSUMES ACTUALS/DRIVERS: directly — prior actuals ARE the baseline; forecast drivers enter only as the uplift %.
  - `spec:` Budget_i = PriorActual_i x (1 + g_i) + Adjustment_i, where g_i = inflation% + growth% + strategic%. Total = sum_i Budget_i.
- **Method 1 worked example (Incremental)** — Prior-year actual marketing spend = $200,000; assume inflation 3% + planned growth 5% = 8% uplift, plus a $15,000 one-off for a new campaign. Budget = 200,000 x 1.08 + 15,000 = 216,000 + 15,000 = $231,000. If next year actual comes in at $225,000, variance = 225,000 - 231,000 = -$6,000 (favorable / under budget), which becomes the new baseline ($225,000 actual) for the following cycle.
  - `spec:` 200000 x (1 + 0.08) + 15000 = 231000
- **Method 2 — Activity-Based Budgeting (ABB): definition, inputs, algorithm, formula, example** — DEFINITION (CFI): a top-down method that 'determines the amount of inputs required to support the targets or outputs set by the company'; Wall Street Mojo: 'the firm first identifies, analyzes, and researches the activities that determine the cost... then prepares the budget based on the results.' WHEN USED: cost-conscious / process-heavy organizations (manufacturing, operations) that want budgets tied to output volumes and operational drivers rather than history; environments where costs scale with measurable activities. INPUTS REQUIRED: output/sales targets; list of activities required to deliver outputs; the cost driver for each activity (e.g., # setups, inspection hours, machine hours); forecast driver VOLUME per activity; cost-driver RATE (cost per driver unit), typically derived from prior actual cost pool / prior driver units. ALGORITHM (autonomous): (1) Set output target (units/revenue). (2) Enumerate activities a needed to hit target. (3) For each activity, identify its cost driver and forecast its volume V_a from the output target (V_a = output x driver-per-output). (4) Compute or load the driver rate R_a = prior cost pool_a / prior driver units_a. (5) Budget_a = V_a x R_a. (6) Sum across activities; add non-activity (fixed) costs. CONSUMES ACTUALS/DRIVERS: heavily — driver RATES come from prior actual cost pools divided by prior actual driver units; driver VOLUMES come from the forecast/output plan.
  - `spec:` Cost Driver Rate R_a = Total Cost Pool_a / Total Cost Driver Units_a. Activity Budget_a = Forecast Driver Volume V_a x R_a. Total Budget = sum_a (V_a x R_a) + fixed costs.
- **Method 2 worked example (ABB)** — Activity = machine setups. Prior actual setup cost pool = $400,000 over 700 setups, so rate R = 400,000 / 700 = $571.43 per setup. Next year's production plan requires 800 setups (forecast driver volume). Setup budget = 800 x 571.43 = $457,144. Second activity = inspection: prior pool $280,000 / 15,500 hours = $18.06/hr; plan needs 16,000 inspection hours -> 16,000 x 18.06 = $288,960. ABB total for these activities = 457,144 + 288,960 = $746,104, versus an incremental approach that would just inflate the prior $680,000 by a flat %.
  - `spec:` R_setup = 400000/700 = 571.43; Budget_setup = 800 x 571.43 = 457144. R_insp = 280000/15500 = 18.06; Budget_insp = 16000 x 18.06 = 288960.
- **Method 3 — Value-Proposition / Priority-Based Budgeting: definition, inputs, algorithm, formula, example** — DEFINITION (Datarails/CFI): a technique ensuring 'every expense included in the budget is delivering value to the business'; it allocates resources by ranking initiatives on value created and funding highest-value first under a constraint. Also called Priority-Based Budgeting. The three CFI test questions per item: Why is this amount included? Does it create value for customers/staff/stakeholders? Does the value outweigh the cost? WHEN USED: growth/strategy-driven organizations allocating a constrained pool across competing initiatives; mindset method sitting between incremental (low rigor) and ZBB (high rigor). INPUTS REQUIRED: a candidate list of expenditures/initiatives; for each: estimated cost, estimated value/benefit (qualitative or quantified — e.g., revenue impact, strategic score), a value scoring rubric, and a total budget constraint (cap). ALGORITHM (autonomous, knapsack/greedy): (1) Enumerate candidate items with cost C_j and value score Val_j. (2) Compute value-to-cost ratio ROI_j = Val_j / C_j. (3) Sort descending by ROI_j (priority ranking). (4) Greedily fund items down the ranked list while cumulative cost <= budget cap (or solve 0/1 knapsack for an optimal mix under the cap). (5) Items below the cutline are deferred/rejected. (6) Output funded set + remaining slack. CONSUMES ACTUALS/DRIVERS: prior actuals and realized ROI feed the value scores and cost estimates; forecast drivers (e.g., projected revenue lift) feed Val_j.
  - `spec:` Priority ratio ROI_j = Value_j / Cost_j. Fund set S maximizing sum(Value_j) subject to sum(Cost_j) <= BudgetCap (0/1 knapsack); greedy approximation = fund by descending ROI_j until cap reached.
- **Method 3 worked example (Value-Proposition)** — Budget cap = $100,000. Candidates: A cost $40k value-score 90 (ROI 2.25); B cost $30k score 75 (2.50); C cost $50k score 60 (1.20); D cost $20k score 50 (2.50). Rank by ratio: B (2.50), D (2.50), A (2.25), C (1.20). Greedy fund: B ($30k, cum 30k), D ($20k, cum 50k), A ($40k, cum 90k); C ($50k) would push cum to 140k > 100k cap, so C is deferred. Funded = {B, D, A} = $90k, total value 215, $10k slack. (A 0/1-knapsack solver could test swapping for higher total value but here greedy is optimal.)
  - `spec:` Rank by Value/Cost; fund descending until sum(Cost) <= 100000. Funded {B,D,A}=90000, deferred {C}.
- **Method 4 — Zero-Based Budgeting (ZBB): definition, inputs, algorithm, formula, example** — DEFINITION (Wikipedia): 'a budgeting method that requires all expenses to be justified and approved in each new budget period'; starts from a zero base with no carry-forward. Originated by Peter Pyhrr at Texas Instruments (HBR 1970); adopted by Georgia (1973) and the US federal government (1977). WHEN USED: cost containment, restructuring, downturns, or periodic deep reviews of discretionary spend; when management wants to break the ratchet of incremental creep. INPUTS REQUIRED: decision UNITS (departments/programs/functions analyzable independently); for each unit, DECISION PACKAGES describing an activity's objective, cost, benefit/justification, alternatives, and typically multiple service/funding levels (minimum/zero-base, current, enhanced); a ranking criterion (cost-benefit); total available funds. ALGORITHM (autonomous): (1) Define decision units. (2) For each unit, generate decision packages at incremental service levels, each with cost and benefit. (3) Score/rank ALL packages across the org by cost-benefit (or value/cost). (4) Allocate funds top-down the ranked list until the funding envelope is exhausted; everything below the cutoff is unfunded. (5) The approved set is the budget — built from zero, not from prior baseline. CONSUMES ACTUALS/DRIVERS: prior actuals/drivers are EVIDENCE used to size each package's cost and quantify its benefit, but are explicitly NOT used as an automatic baseline — every dollar must be re-justified.
  - `spec:` For each decision package p: net value N_p = Benefit_p - Cost_p (or ratio Benefit_p/Cost_p). Rank all p descending; fund while cumulative Cost_p <= TotalFunds. Budget = sum(Cost_p for funded p). Packages offered at levels L (min/current/enhanced).
- **Method 4 worked example (ZBB)** — Decision unit = Customer Support. Total funds available = $250,000. Packages: P1 'Core ticket handling, minimum level' cost $120k benefit-score 100; P2 'Add live chat' cost $60k benefit 70; P3 'Add 24/7 coverage' cost $90k benefit 40; P4 'Premium onboarding team' cost $80k benefit 25. Rank by benefit/cost: P1 (0.83), P2 (1.17), P3 (0.44), P4 (0.31) -> order P2, P1, P3, P4. Fund: P2 ($60k cum 60k), P1 ($120k cum 180k), P3 ($90k would be cum 270k > 250k) so P3 partially or deferred; with strict 0/1: fund P2+P1 = $180k, then P4 ($80k) cum 260k > 250 deferred; remaining $70k could fund a scaled P3. Final approved budget rebuilt from zero = $180k core + scaled add-ons up to $250k cap, with every funded package individually justified.
  - `spec:` Rank by Benefit/Cost; fund cumulatively <= 250000. P2(60k)+P1(120k)=180k funded; P3/P4 compete for remaining 70k.
- **Static vs Flexible vs Rolling (the competing list) — treat as MODES not methods** — STATIC budget (AccountingTools): 'prepared for a single expected level of activity and is not adjusted after the period begins' — based on one projected volume. FLEXIBLE budget: 'adjusts budgeted revenues and costs to match the actual activity level achieved'; budgeted variable costs are recomputed at ACTUAL volume, isolating volume variance from efficiency/price variance. ROLLING/CONTINUOUS budget: 'continuously updated by adding a new period as the most recent period is completed,' keeping a constant horizon (e.g., always 12 months forward). These describe budget BEHAVIOR over volume/time, not how line items are justified, so they can wrap any of the four derivation engines. Implementation: Flexible mode re-evaluates variable lines as VariableCost = StandardRate x ActualVolume + FixedCost; Rolling mode is a scheduler that re-runs the chosen engine each period and drops the oldest period while appending a new one.
  - `spec:` Flexible budget allowance = (Std variable cost per unit x Actual units) + Budgeted fixed cost. Rolling: at period close, horizon = [t+1 ... t+N]; re-run engine; N constant.
- **How each method consumes prior actuals and forecast drivers (product data-flow)** — Common input layer the tool should expose to all four engines: (a) ACTUALS store — historical GL/cost-center spend by period and line; (b) DRIVERS store — operational/forecast drivers (volume, headcount, sales target, inflation%, FX, activity counts). Mapping: INCREMENTAL = actuals as baseline x driver(inflation/growth) uplift. ABB = actuals used to compute driver rates (cost pool/units), forecast drivers supply future volumes; rate x volume. VALUE-PROPOSITION = actuals/realized-ROI and forecast revenue-lift feed value scores; cost estimates from actuals; constraint optimization. ZBB = actuals/drivers are justification evidence sizing each decision package's cost and quantifying benefit, never an automatic carry-forward. Thus a single (actuals + drivers) data model serves all four; the engines differ only in HOW they transform that data into line amounts.
  - `spec:` Shared schema: Actuals[period, costCenter, account, amount]; Drivers[period, driverName, value, unit]. Engine(method) -> Budget[period, costCenter, account, amount, justification?].

**Recommendations**

- Implement the canonical four derivation engines as the product's 'budgeting methods': INCREMENTAL, ACTIVITY_BASED (ABB), VALUE_PROPOSITION (priority-based), and ZERO_BASED (ZBB). This matches the most-cited 'four types of budgets' (CFI) and each maps to a distinct deterministic algorithm.
- Do NOT make Static/Flexible/Rolling compete as alternatives to the four. Model them as orthogonal modifiers: a volumeMode toggle {STATIC, FLEXIBLE} that re-evaluates variable lines at actual volume, and a cadence setting {PERIODIC, ROLLING(horizon)} implemented as a scheduler. This lets users combine e.g. 'Rolling ABB' or 'Flexible Incremental' and resolves the two competing lists cleanly.
- Build a single shared input layer: an Actuals store (period x cost center x account) and a Drivers store (volumes, inflation%, growth%, FX, activity counts). All four engines read from this; only the transformation differs. Document the per-engine data flow so the build team knows Incremental/ABB consume actuals mechanically while ZBB/Value-Proposition consume them as justification evidence.
- Encode the formulas exactly as specified: Incremental = PriorActual x (1+g) + adj; ABB = (CostPool/DriverUnits) x ForecastVolume; Value-Proposition = greedy/knapsack fund by descending Value/Cost under a cap; ZBB = rank decision packages by Benefit/Cost and fund cumulatively to the funding envelope. Include the worked numeric examples as unit-test fixtures.
- For Value-Proposition and ZBB, implement a configurable scoring rubric and a constraint optimizer (greedy for speed, optional 0/1 knapsack solver for optimal allocation under a hard cap) plus a decision-package data model (objective, cost, benefit, alternatives, service levels) so the methods can run autonomously once a user supplies candidates and a budget cap.
- After each cycle, run automated variance analysis (Actual - Budget, with flexible-budget variance decomposition into volume vs rate) and feed results back: as the next baseline for Incremental, as refreshed driver rates for ABB, and as realized-ROI evidence for Value-Proposition/ZBB scoring.
- Note licensing: all sources consulted are reference/educational web content (CFI, AccountingTools, Wikipedia, Wall Street Mojo, university LibreTexts/Lumen). The methods themselves are non-proprietary management-accounting techniques and free to implement; no library license constraints apply to the algorithms.

**Open questions**

- Should the product expose Static/Flexible/Rolling as user-visible modifiers, or hide them behind sensible defaults (e.g., always rolling 12-month, always flexible for variable lines)? This is a UX/product decision the build team must confirm.
- For Value-Proposition and ZBB, what value/benefit scoring inputs will users realistically supply (pure manual scores, revenue-impact estimates, or a weighted strategic rubric)? The optimizer's usefulness depends on the quality of these inputs, which may need a guided wizard.
- Does the target customer base lean operational/manufacturing (favoring ABB with rich driver data) or services/SaaS (favoring Incremental + Value-Proposition)? This affects which engine to make the default and how much to invest in the cost-driver modeling subsystem.
- ZBB at full rigor is very labor-intensive; should the tool offer a 'lite' ZBB (re-justify only discretionary/flagged categories above a threshold) to make it practical, and how is that threshold configured?
- Investopedia's ZBB page could not be fetched (blocked); the ZBB definition here is sourced from Wikipedia, AccountingTools, and eFinanceManagement instead — confirm whether a formal accounting-body definition (e.g., CIMA/IMA) is required for product documentation/compliance.

### forecasting-methods
Research covers three layers for an autonomous FP&A forecasting engine. (a) Classic FP&A methods: straight-line/growth-rate (compound projection from a single growth rate), moving average (rolling n-period mean, smooths noise), simple linear regression (one driver, Y=mX+c), and multiple linear regression (several drivers) — all implementable with Excel-equivalent math and trivially in numpy/statsmodels. (b) Statistical time-series models for autonomy: ARIMA/SARIMA (captures autocorrelation + trend via differencing + seasonality via seasonal order), Exponential Smoothing / ETS (Holt = trend, Holt-Winters = trend+seasonality, ETS = full error/trend/seasonal taxonomy with state-space), and Prophet (decomposable trend + multiple seasonalities + holidays, robust to gaps/outliers). Each has a concrete Python API: statsmodels SARIMAX/ExponentialSmoothing/ETSModel, pmdarima auto_arima, Nixtla statsforecast AutoARIMA/AutoETS/AutoTheta/AutoCES (fastest, built-in prediction intervals + cross_validation), and prophet Prophet. (c) Autonomy is achieved by fitting a candidate pool per series, scoring each with rolling-origin (time-series) cross-validation, ranking by a robust error metric (MASE recommended; sMAPE/RMSE/MAE/MAPE as alternates), picking the lowest-error model per series, detecting seasonality via ACF/periodogram (or letting auto-models test via KPSS/OCSB/Canova-Hansen), falling back to naive/seasonal-naive/simple-ES for short series (need more obs than parameters), and emitting prediction intervals (model-based normal intervals or distribution-free conformal). Recommended stack for the product: Nixtla statsforecast (Apache-2.0) as the autonomous core, with Prophet (MIT) for messy business calendars; statsmodels (BSD-3) and pmdarima (MIT) as supporting libraries.

**Key findings**

- **Straight-line / growth-rate method (classic FP&A)** — Simplest forecast: assumes a constant period-over-period growth rate derived from history and compounds it forward. Growth rate g = (Current period - Prior period) / Prior period. Future value = Previous value x (1 + g). Inputs: a single historical series and one assumed constant growth rate (or the average of historical growth rates). Worked example: 2024 revenue = 1,000,000; observed YoY growth g = 12% => 2025 forecast = 1,000,000 x 1.12 = 1,120,000; 2026 = 1,120,000 x 1.12 = 1,254,400. Excel: growth =(C7-B7)/B7, forecast =C7*(1+D5). Pros: trivial, transparent. Cons: ignores seasonality, cycles, and any non-linearity; assumes the past growth persists. Best for stable, mature lines and long-horizon top-line planning.
  - `spec:` g = (V_t - V_{t-1}) / V_{t-1};  Forecast_{t+1} = V_t * (1 + g);  Excel: =(C7-B7)/B7 and =C7*(1+D5)
- **Moving average method (classic FP&A)** — Forecasts the next period as the arithmetic mean of the last n observations, smoothing short-term peaks and troughs to reveal the underlying level/trend. Common windows: 3-month, 5-month, or quarterly. Inputs: historical series and a chosen window length n (smaller n = more responsive, larger n = smoother). Worked example with a 3-month window on revenue [100, 110, 120, 130]: forecast for period 5 = (110+120+130)/3 = 120. Excel: =AVERAGE(B4:B6). Variants: simple MA, weighted MA (recent periods weighted more), and exponential smoothing (a recursive weighted MA). Cons: lags real turning points, cannot extrapolate trend (flat-ish forecasts), no seasonality unless paired with seasonal indices.
  - `spec:` MA_t = (1/n) * sum_{i=0}^{n-1} y_{t-i};  Forecast_{t+1} = MA_t;  Excel: =AVERAGE(B4:B6)
- **Simple linear regression (one driver)** — Fits a straight line relating a dependent variable Y to one independent variable X (often time or a single business driver such as marketing spend or headcount). Model: Y = mX + c + e, where m is slope, c is intercept, e is residual error; coefficients estimated by ordinary least squares (minimizing sum of squared residuals). Inputs: paired observations (X_i, Y_i). Also yields confidence intervals and p-values for statistical significance. CFI worked example: slope m = 78.08, intercept c = 7930.35 => for a given X, Forecast = 78.08*X + 7930.35 (e.g., X=20 => 78.08*20 + 7930.35 = 9491.95). Excel: m=SLOPE(Y,X), c=INTERCEPT(Y,X), predict =FORECAST(x,Y,X) or =(SLOPE*x)+INTERCEPT; LINEST returns full stats. Python: numpy.polyfit(x,y,1) or statsmodels.api.OLS / scipy.stats.linregress.
  - `spec:` Y = m*X + c + e;  m = Σ(x-x̄)(y-ȳ)/Σ(x-x̄)^2;  c = ȳ - m*x̄;  Excel: =SLOPE(), =INTERCEPT(), =FORECAST(x,knownY,knownX); Python: numpy.polyfit / statsmodels OLS
- **Multiple linear regression (several drivers)** — Generalizes simple regression to two or more independent variables predicting one dependent variable, capturing the combined effect of multiple drivers. Model: Y = b0 + b1*X1 + b2*X2 + ... + bk*Xk + e, coefficients by OLS. Inputs: a matrix of driver observations plus the target. CFI Excel route: Data tab > Data Analysis > Regression, supply Y range and X ranges; forecast with =b0 + X1*b1 + X2*b2 (Excel =$B$25+(B18*$B$26)+(B19*$B$27)). Cautions for autonomy: requires future values of the drivers (themselves forecast or assumed), watch for multicollinearity, overfitting with too many regressors, and spurious correlation. Python: statsmodels.api.OLS(y, sm.add_constant(X)).fit() gives coefficients, R^2, p-values, and prediction intervals via get_prediction().
  - `spec:` Y = b0 + b1*X1 + ... + bk*Xk + e (OLS);  Excel: Data Analysis > Regression, =$B$25+(B18*$B$26)+(B19*$B$27); Python: statsmodels.api.OLS(y, sm.add_constant(X)).fit()
- **ARIMA / SARIMA — model and what it captures** — ARIMA(p,d,q): AR(p) regresses on p past values, I(d) differences the series d times to remove trend/achieve stationarity, MA(q) regresses on q past forecast errors. SARIMA adds a seasonal block (P,D,Q,s): seasonal AR/diff/MA at lag s (s=12 monthly, 4 quarterly, 7 daily-with-weekly). Captures autocorrelation, trend (via d), and seasonality (via seasonal order). Required data: a reasonably long, ideally regularly-spaced univariate series; for reliable seasonal estimation prefer >= 2 full seasonal cycles (e.g., >=24 monthly points for s=12). SARIMAX additionally accepts exogenous regressors (drivers) = 'regression with SARIMA errors'. Key params: order=(p,d,q), seasonal_order=(P,D,Q,s), trend.
  - `spec:` ARIMA(p,d,q)(P,D,Q)_s. statsmodels: SARIMAX(endog, exog=None, order=(p,d,q), seasonal_order=(P,D,Q,s), trend=None, enforce_stationarity=True, enforce_invertibility=True). statsmodels v0.14.6
- **ARIMA/SARIMA — exact Python APIs (statsmodels, pmdarima)** — statsmodels manual fit: from statsmodels.tsa.statespace.sarimax import SARIMAX; res = SARIMAX(data, order=(1,1,1), seasonal_order=(1,1,1,12)).fit(); fc = res.get_forecast(steps=12); fc.predicted_mean and fc.conf_int(alpha=0.05) for 95% intervals. Automatic order selection with pmdarima: from pmdarima import auto_arima; model = auto_arima(y, seasonal=True, m=12, stepwise=True, information_criterion='aic'); model.predict(n_periods=12, return_conf_int=True). pmdarima implements the Hyndman-Khandakar stepwise search; m = periods per season (1=non-seasonal, 4 quarterly, 12 monthly); d chosen by KPSS test (default), D by OCSB test (default; alternatives Canova-Hansen 'ch', 'hegy', 'seas'); minimizes AIC/AICc/BIC/HQIC/OOB. pmdarima 2.1.1.
  - `spec:` auto_arima(y, seasonal=True, m=12, stepwise=True, information_criterion='aic', test='kpss', seasonal_test='ocsb').predict(n_periods=h, return_conf_int=True). pmdarima 2.1.1; statsmodels SARIMAX.get_forecast(steps=h).conf_int(alpha=0.05)
- **Exponential Smoothing — Holt, Holt-Winters, ETS taxonomy** — Simple Exponential Smoothing (SES): level only, no trend/seasonality, forecast = weighted average with weights decaying exponentially (smoothing param alpha). Holt's linear method: adds a trend component (level alpha + trend beta), optionally damped (phi). Holt-Winters: adds a seasonal component (gamma), in additive form (seasonal amplitude constant) or multiplicative form (seasonal amplitude scales with level); seasonal_periods m. ETS generalizes these into an Error-Trend-Seasonal taxonomy: Error in {add, mul}, Trend in {none, add, add-damped, mul, mul-damped}, Seasonal in {none, add, mul} — each combination is an ETS(E,T,S) model with an underlying state-space form that also yields prediction intervals. Required data: SES/Holt work on short series; Holt-Winters/seasonal ETS need >= 2 full seasonal cycles. Captures: trend (Holt/ETS-T) and seasonality (Holt-Winters/ETS-S).
  - `spec:` Holt-Winters additive: level l_t=α(y_t - s_{t-m})+(1-α)(l_{t-1}+b_{t-1}); trend b_t=β(l_t-l_{t-1})+(1-β)b_{t-1}; seasonal s_t=γ(y_t-l_{t-1}-b_{t-1})+(1-γ)s_{t-m}; forecast ŷ_{t+h}=l_t+h·b_t+s_{t+h-m}. ETS(E,T,S) with E∈{A,M}, T∈{N,A,Ad,M,Md}, S∈{N,A,M}
- **Exponential Smoothing — exact Python APIs (statsmodels)** — Holt-Winters (classic): from statsmodels.tsa.holtwinters import ExponentialSmoothing; fit = ExponentialSmoothing(y, trend='add', seasonal='add', seasonal_periods=12, damped_trend=False, initialization_method='estimated').fit(); fit.forecast(12). Parameters trend in {None,'add','mul'}, seasonal in {None,'add','mul'}, seasonal_periods=m; fit() can take smoothing_level (alpha), smoothing_trend (beta), smoothing_seasonal (gamma), damping_trend (phi) or estimate them via MLE (optimized=True). State-space ETS with prediction intervals and automatic component options: from statsmodels.tsa.exponential_smoothing.ets import ETSModel; ETSModel(y, error='add', trend='add', damped_trend=True, seasonal='add', seasonal_periods=12).fit(); use .get_prediction(start,end).summary_frame() for intervals. statsmodels v0.14.6.
  - `spec:` ExponentialSmoothing(y, trend='add', seasonal='add', seasonal_periods=12, damped_trend=False, initialization_method='estimated').fit().forecast(h); ETSModel(y, error='add', trend='add', seasonal='add', seasonal_periods=12).fit().get_prediction(...).summary_frame(). statsmodels 0.14.6
- **Prophet — model, capabilities, exact API** — Additive (or multiplicative) decomposable model: y(t) = trend g(t) + seasonality s(t) + holidays h(t) + error. Captures piecewise-linear or logistic trend with automatic changepoints, multiple seasonalities (yearly/weekly/daily by default, plus custom) via Fourier series, and holiday/event effects; robust to missing data and outliers and tolerant of irregular spacing. Required data: a dataframe with two columns, ds (datestamp YYYY-MM-DD or with time) and y (numeric). API: from prophet import Prophet; m = Prophet(interval_width=0.8, seasonality_mode='additive', changepoint_prior_scale=0.05); m.fit(df); future = m.make_future_dataframe(periods=365, freq='D'); forecast = m.predict(future) -> columns yhat, yhat_lower, yhat_upper. Custom seasonality via m.add_seasonality(name, period, fourier_order, mode); holidays via add_country_holidays or a holidays dataframe. Key knobs: interval_width (default 0.8 = 80% interval), seasonality_mode ('additive'/'multiplicative'), changepoint_prior_scale (default 0.05; higher = more flexible trend). pip install prophet.
  - `spec:` y(t)=g(t)+s(t)+h(t)+ε(t). Prophet(interval_width=0.8, seasonality_mode='additive', changepoint_prior_scale=0.05); m.fit(df[['ds','y']]); m.predict(m.make_future_dataframe(periods=h)) -> yhat/yhat_lower/yhat_upper. pip install prophet
- **Nixtla statsforecast — autonomous model pool, exact API** — Fastest path to autonomous multi-series forecasting; scikit-learn-style fit/predict, optimized in numba, scales to thousands of series. Input long dataframe with columns unique_id, ds, y. Models: from statsforecast.models import AutoARIMA, AutoETS, AutoTheta, AutoCES (also AutoTBATS, SeasonalNaive, Naive, WindowAverage). Setup: from statsforecast import StatsForecast; sf = StatsForecast(models=[AutoARIMA(season_length=12), AutoETS(season_length=12), AutoTheta(season_length=12), AutoCES(season_length=12)], freq='M', n_jobs=-1). Because all candidate models run in one call, statsforecast is ideal as the engine that fits many models per series, then you pick the winner via its cross-validation.
  - `spec:` sf = StatsForecast(models=[AutoARIMA(season_length=m), AutoETS(season_length=m), AutoTheta(season_length=m), AutoCES(season_length=m)], freq=..., n_jobs=-1); sf.forecast(df=Y_df, h=h, level=[80,90]). Input cols: unique_id, ds, y. pip install statsforecast
- **Autonomy — rolling-origin (time-series) cross-validation** — Do NOT use random k-fold (it leaks future into past). Use evaluation on a rolling forecasting origin: the origin moves forward in time; for each window train only on data up to the cutoff and forecast h steps; accuracy is computed on each test window and averaged across windows. This simulates real recurring forecasting and gives an honest estimate of out-of-sample error. statsforecast implements it directly: cv_df = sf.cross_validation(df=Y_df, h=12, step_size=12, n_windows=3) -> output columns unique_id, ds, cutoff, y, and one prediction column per model; group by model, compute the error metric, and select the model with the lowest score. (sktime/Darts offer equivalent ExpandingWindowSplitter / historical_forecasts.) Guidance: each test window/horizon should be at least as large as the operational forecast horizon, and the test span ideally ~20% of data.
  - `spec:` sf.cross_validation(df, h, step_size, n_windows) -> cols [unique_id, ds, cutoff, y, <model>]; expanding/rolling origin: train on y[:cutoff], forecast h, roll cutoff by step_size, repeat n_windows; average per-window error
- **Autonomy — error metrics (formulas) and which to pick** — Let e_t = y_t - ŷ_t. MAE = mean(|e_t|) (scale-dependent, robust). RMSE = sqrt(mean(e_t^2)) (scale-dependent, penalizes large errors; minimized by the mean). MAPE = mean(|e_t|/|y_t|)*100 (scale-free but undefined/explosive near y_t=0 and asymmetric — penalizes over-forecasts more). sMAPE = mean(200*|y_t-ŷ_t|/(|y_t|+|ŷ_t|)) (used in M3 but can go negative and is unstable near zero; Hyndman & Koehler recommend against it). MASE = mean(|e_t|)/Q where Q is the in-sample MAE of a benchmark: non-seasonal Q = (1/(T-1))*sum_{t=2..T}|y_t - y_{t-1}| (naive); seasonal Q = (1/(T-m))*sum_{t=m+1..T}|y_t - y_{t-m}| (seasonal naive). MASE is scale-invariant, defined for zero/intermittent data, and <1 means the model beats naive — the recommended primary metric for autonomous model selection across heterogeneous FP&A series. Use RMSE or MAE as the secondary tie-breaker; report MAPE only for business readability where y is comfortably positive.
  - `spec:` MAE=mean(|e_t|); RMSE=sqrt(mean(e_t^2)); MAPE=100*mean(|e_t/y_t|); sMAPE=mean(200|y_t-ŷ_t|/(|y_t|+|ŷ_t|)); MASE=mean(|e_t|)/Q, Q_nonseasonal=(1/(T-1))Σ|y_t-y_{t-1}|, Q_seasonal=(1/(T-m))Σ|y_t-y_{t-m}|
- **Autonomy — automatic model selection workflow** — Per series: (1) clean & regularize (resample to fixed freq, fill/flag gaps); (2) detect frequency and seasonal period m; (3) build a candidate pool sized to the data (e.g., SeasonalNaive, Naive, WindowAverage, AutoETS, AutoARIMA, AutoTheta, AutoCES, optionally Prophet); (4) run rolling-origin CV over the pool with horizon = business horizon and several windows; (5) score each candidate with MASE (plus RMSE tie-break) averaged across windows; (6) select the lowest-error model for that series; (7) refit the winner on the full history and forecast h with prediction intervals. The auto-models (AutoARIMA/AutoETS) already internally minimize AICc to pick their own orders/components, so CV is the outer selector across model families and AIC is the inner selector within a family. This is exactly the design Nixtla promotes (run many automatic models, compare via cross_validation, choose lowest error).
  - `spec:` for each series: candidates -> sf.cross_validation(h, step_size, n_windows) -> aggregate MASE per model -> argmin -> refit winner on full history -> forecast(h, level=[80,95]). Inner: AutoARIMA/AutoETS minimize AICc; Outer: CV minimizes MASE
- **Autonomy — seasonality detection** — Two standard automatic approaches: (1) ACF-based — compute the autocorrelation function up to a max lag, find peaks (a lag whose ACF exceeds its neighbors and the white-noise significance band); the seasonal period is typically the strongest peak among positive lags following the first run of negative correlations. (2) Periodogram / FFT — estimate the power spectral density and take the dominant frequency, whose inverse is the period. In practice, for calendar data set m by frequency a priori (daily=7, monthly=12, quarterly=4, weekly=52) and let auto-models confirm whether seasonal differencing/components are warranted: pmdarima/AutoARIMA decide seasonal D via OCSB (default) or Canova-Hansen tests; AutoETS decides whether the S component improves AICc. Strength-of-seasonality (Fs from STL decomposition) can gate whether to even attempt seasonal models.
  - `spec:` ACF peak detection: period = argmax_k ACF(k) over significant positive lags; Periodogram: period = 1/f* where f*=argmax PSD(f); seasonal-diff decision via OCSB/Canova-Hansen (auto_arima seasonal_test)
- **Autonomy — short-series fallback** — Theoretical floor: you need more observations than the model has parameters; in practice substantially more, and AIC on short series will favor very simple models because anything richer overfits. Practical rules: require >= 2 full seasonal cycles (e.g., >=24 monthly points) before allowing seasonal models (SARIMA/Holt-Winters/seasonal ETS); below that, drop to non-seasonal models (Holt, SES, non-seasonal ARIMA). For very short series, fall back to deterministic baselines: Naive (forecast = last value — strong for many financial/random-walk series), Seasonal Naive (forecast = value from one season ago), drift, mean/window-average, or simple exponential smoothing. Industry thresholds (e.g., Microsoft Dynamics) switch to naive when the series is below a bucket-dependent length (14 days / 12 weeks / 12 months) and no seasonality is detected. These naive methods also serve as the MASE benchmark denominator, so they double as the accuracy yardstick.
  - `spec:` Naive: ŷ_{t+h}=y_t; Seasonal Naive: ŷ_{t+h}=y_{t+h-m}; Drift: ŷ_{t+h}=y_t + h*(y_t-y_1)/(t-1). Gate: seasonal models only if n >= 2*m; thresholds ~14d/12w/12m
- **Autonomy — prediction intervals** — Two production options. (1) Model-based (parametric): state-space models give analytic intervals — statsmodels SARIMAX.get_forecast(h).conf_int(alpha=0.05); ETSModel.get_prediction(...).summary_frame(); pmdarima predict(return_conf_int=True); Prophet returns yhat_lower/yhat_upper governed by interval_width (default 0.8). These assume (often Gaussian) error distributions and can be mis-calibrated for skewed/heteroscedastic financial data. (2) Distribution-free conformal prediction: wrap any model and calibrate intervals on out-of-sample residuals so coverage is guaranteed without distributional assumptions; statsforecast supports this — pass level=[80,90,95] to forecast()/cross_validation() to emit nested <model>-lo-XX / <model>-hi-XX columns (the 80% interval nests inside 90% inside 95%). For an autonomous FP&A tool, prefer conformal/empirically-calibrated intervals (validate coverage on the CV windows) and always surface lower/upper bands alongside the point forecast.
  - `spec:` Parametric: SARIMAX.get_forecast(h).conf_int(alpha); ETSModel.get_prediction().summary_frame(); Prophet interval_width=0.8 -> yhat_lower/upper. Conformal: sf.forecast(h, level=[80,90,95]) -> <model>-lo/hi-XX, calibrated on residuals

  - `spec:` statsmodels=BSD-3-Clause; pmdarima=MIT; statsforecast=Apache-2.0; prophet=MIT

**Recommendations**

- Adopt a two-tier model menu: (Tier 1, explainable for finance users) straight-line growth, moving average, simple/multiple linear regression — keep these because FP&A teams trust and audit them; (Tier 2, autonomous) statistical time-series models that the engine selects automatically. Expose both so users can override the auto-pick.
- Use Nixtla statsforecast (Apache-2.0) as the autonomous core: it runs AutoARIMA + AutoETS + AutoTheta + AutoCES + SeasonalNaive/Naive in one call across many series, has built-in cross_validation, and emits prediction intervals (including distribution-free conformal). Add Prophet (MIT) only for series with strong holiday/calendar effects or messy gaps.
- Make selection autonomous via rolling-origin cross-validation, not random k-fold: sf.cross_validation(h=<business horizon>, step_size, n_windows>=3), aggregate MASE per model across windows, and pick the lowest-MASE model per series; refit the winner on full history before producing the live forecast.
- Standardize on MASE as the primary accuracy metric (scale-invariant, handles zeros/intermittent demand, <1 beats naive) with RMSE or MAE as a tie-breaker. Show MAPE in the UI only for clearly positive series; avoid sMAPE for ranking (Hyndman & Koehler advise against it).
- Implement an explicit data-sufficiency gate: require >= 2 full seasonal cycles (e.g., >=24 monthly points) before allowing SARIMA/Holt-Winters/seasonal-ETS; for shorter series fall back through Holt -> SES -> Seasonal Naive -> Naive. These naive baselines double as the MASE benchmark denominator.
- Set the seasonal period m by calendar frequency a priori (12 monthly, 4 quarterly, 7/52 for daily/weekly) and let auto-models confirm seasonality (OCSB/Canova-Hansen for ARIMA seasonal differencing, AICc for ETS components); optionally add an ACF/periodogram detector for non-calendar periods.
- Always return prediction intervals, not just point forecasts. Prefer conformal intervals calibrated on the CV residuals (validate empirical coverage against the nominal level), and surface lower/upper bands at e.g. 80% and 95% in the UI.
- Cache per-series the winning model family, its hyperparameters, the CV error, and the chosen m; re-run model selection on a schedule (e.g., monthly) but re-fit the cached winner more frequently to keep latency low. Log the CV scoreboard for auditability/explainability to FP&A stakeholders.
- For driver-based (regression) forecasts, remember the drivers must themselves be forecast or supplied as assumptions; support scenario inputs and guard against multicollinearity/overfitting. statsmodels OLS / SARIMAX-with-exog covers this cleanly with analytic intervals.
- Pin and test library versions in CI: statsforecast 2.x, statsmodels 0.14.x, prophet 1.1.x (Stan/cmdstanpy build), pmdarima 2.1.x (watch NumPy/SciPy compatibility). All licenses (Apache-2.0 / BSD-3 / MIT) are commercial-safe; keep an attributions/NOTICE file.

### variance-analysis
Variance analysis decomposes the gap between actual results and plan into named, additive drivers (price, rate, efficiency, mix, volume, spending). The master hierarchy is: Static (master) Budget Variance = Flexible Budget Variance + Sales Volume Variance. The flexible budget restates the plan at ACTUAL output (variable items flexed per-unit, fixed costs unchanged), so the flexible-budget variance isolates price/rate/efficiency effects and the sales-volume variance isolates the quantity effect. The single most error-prone area is SIGN CONVENTION. The robust, sign-safe rule used across authoritative texts (Saylor/Lumen): compute every variance as (Actual cost incurred) - (Standard/budgeted cost allowed), i.e. left-to-right Actual minus Standard. For COSTS, a positive result is UNFAVORABLE (spent more) and negative is FAVORABLE. For REVENUE/CONTRIBUTION/PROFIT the labeling flips: a positive result (actual revenue > budget) is FAVORABLE, negative is UNFAVORABLE. Best practice in a product: store the signed numeric variance with an explicit favorable/unfavorable flag derived from the account TYPE (revenue vs cost), never infer F/U from sign alone. Cost variance formulas: Material Price = (AP-SP)xAQ purchased; Material Quantity = (AQ used - SQ allowed)xSP; Labor Rate = (AR-SR)xAH; Labor Efficiency = (AH-SH)xSR; VOH Spending = (AR-SR)xAH; VOH Efficiency = (AH-SH)xSR; Fixed OH Spending/Budget = Actual FOH - Budgeted FOH; Fixed OH Volume = Budgeted FOH - Applied FOH (= std FOH rate/unit x (budgeted units - actual units)). Sales variances: Sales Price = (AP-BP)xActual units; Sales Volume = (Actual units - Budgeted units)x budgeted CM/unit (or budgeted price, or budgeted profit/unit depending on costing basis); for multiproduct, Sales Volume splits into Sales Mix (effect of changing proportions) and Sales Quantity (effect of total units at constant budgeted mix). Presentation: variance bridge/waterfall (Budget -> Volume -> Mix -> Price -> Cost effects -> Actual), percentage thresholds for flagging (commonly material if |abs| > a dollar floor AND |%| > 5-10%), and drill-down by dimension (entity, product, cost center, account). Sources are authoritative open textbooks (Saylor Managerial Accounting Ch.10, Lumen, AccountingTools, AccountingForManagement, accounting-simplified) plus FP&A practitioner sources for PVM waterfalls.

**Key findings**

- **Master variance hierarchy: Static Budget Variance = Flexible Budget Variance + Sales Volume Variance** — The total gap between actual and the original master (static) budget decomposes into two parts. The FLEXIBLE BUDGET restates the master budget at the ACTUAL activity level: variable revenues and variable costs are recomputed at budgeted per-unit rates times actual units; fixed costs stay at the original budgeted amount. (1) Static (master) budget variance = Actual results - Static budget. (2) Flexible budget variance = Actual results - Flexible budget (isolates price, rate, efficiency, spending effects because volume is held at actual). (3) Sales volume (activity) variance = Flexible budget - Static budget (isolates the effect of selling/producing a different number of units, holding per-unit prices and costs at budget). These satisfy: Static budget variance = Flexible budget variance + Sales volume variance. Worked example (Lumen, actual volume 180,000 units): for Sales Revenue the static-budget variance was +168,230 F, which split into sales-volume variance +258,230 F and flexible-budget variance -90,000 U — i.e. the favorable static total masked an unfavorable price effect once flexed to actual volume.
  - `spec:` FlexBudget(item)=BudgetedRatePerUnit x ActualUnits (variable items); =BudgetedAmount (fixed items). FlexBudgetVar=Actual-Flex; SalesVolumeVar=Flex-Static; StaticVar=Actual-Static=FlexBudgetVar+SalesVolumeVar
- **Sign convention and favorable/unfavorable labeling (the error-prone core)** — Use a single consistent arithmetic rule for ALL cost variances: Variance = (Actual quantity x Actual price/rate) - (Allowed quantity x Standard price/rate), i.e. Actual MINUS Standard, left to right. Saylor states the rule explicitly: 'all positive variances are unfavorable, and all negative variances are favorable' — this is true ONLY for cost/expense items. For revenue, contribution margin, or profit items the polarity inverts: actual exceeding budget (positive) is FAVORABLE; actual below budget (negative) is UNFAVORABLE. FAVORABLE means the variance increases operating income (lower cost OR higher revenue); UNFAVORABLE means it decreases operating income (higher cost OR lower revenue). PRODUCT DESIGN RECOMMENDATION: persist the signed numeric variance plus a boolean is_favorable computed from account_type: for cost accounts is_favorable = (variance < 0); for revenue/CM/profit accounts is_favorable = (variance > 0). Do NOT label F/U from the sign alone without knowing whether the line is a revenue or a cost. Note favorable is not automatically 'good' — e.g. a favorable material price variance from buying cheap, low-grade material can cause an unfavorable usage variance (interrelated variances).
  - `spec:` CostVar=Actual-Standard: >0 => Unfavorable, <0 => Favorable. RevenueVar=Actual-Budget: >0 => Favorable, <0 => Unfavorable. is_favorable = (account is cost) ? (var<0) : (var>0)
- **Budget vs Actual variance: absolute and percentage** — Absolute variance = Actual - Budget (in currency). Percentage variance = (Actual - Budget) / Budget x 100, expressed against the budget base (use absolute value of the denominator to keep the sign meaningful when budget is negative; guard against budget = 0 which makes % undefined/infinite — flag as N/A or use a dollar-only threshold). For costs, a positive absolute variance is unfavorable; for revenue it is favorable. Common FP&A presentation shows Budget, Actual, $ Variance, % Variance, and an F/U flag side by side. Percentage variance is dimensionless and used for cross-line comparability and threshold flagging; absolute variance is used for materiality in dollar terms.
  - `spec:` Abs Var = Actual - Budget; % Var = (Actual - Budget) / |Budget| x 100 (undefined when Budget=0)
- **Sales Price Variance** — Isolates the revenue effect of charging a different selling price than budgeted. Formula: (Actual selling price per unit - Budgeted/standard selling price per unit) x Actual units sold. It uses ACTUAL units (not budgeted) so the pricing effect is cleanly separated from the volume effect. Favorable when actual price > budgeted price (positive result is favorable because this is a revenue item); unfavorable when actual price < budgeted price. Worked example: budgeted price 10.00, actual price 10.50, actual units 4,835,000 => (10.50-10.00) x 4,835,000 = +2,417,500 Favorable. Terms: Actual price = realized average selling price per unit; Budgeted price = planned price per unit; Actual units sold = units actually sold in the period.
  - `spec:` Sales Price Variance = (Actual Price - Budgeted Price) x Actual Units Sold
- **Sales Volume Variance** — Isolates the effect of selling a different NUMBER of units than budgeted, holding per-unit economics at budget. Three valid multiplier bases depending on costing approach: (a) budgeted CONTRIBUTION MARGIN per unit (variable/marginal costing — most common in management accounting and recommended for profit-focused analysis), (b) budgeted PROFIT per unit (absorption costing), or (c) budgeted PRICE per unit (when measuring the revenue-only volume effect). Formula: (Actual units sold - Budgeted units sold) x budgeted CM (or profit, or price) per unit. Favorable when actual units > budgeted units. Worked example (revenue basis): budgeted 5,000,000 units, sold 4,835,000, budgeted price 1.15 => (4,835,000 - 5,000,000) x 1.15 = -189,750 Unfavorable. IMPORTANT: pick ONE multiplier basis and use it consistently; the contribution-margin basis ties the sales-volume variance back to the operating-income bridge cleanly.
  - `spec:` Sales Volume Variance = (Actual Units - Budgeted Units) x Budgeted CM per Unit (or budgeted profit/unit, or budgeted price/unit)
- **Sales Mix Variance vs Sales Quantity Variance (multiproduct split of Sales Volume Variance)** — For multiproduct firms the sales VOLUME variance splits into MIX and QUANTITY. SALES MIX VARIANCE measures the profit effect of selling products in different PROPORTIONS than budgeted, holding total units at actual: for each product, (Actual units sold of product - units of that product if actual TOTAL units had been split in the BUDGETED mix) x budgeted CM per unit. Equivalently per product: Actual total units x (Actual mix% - Budgeted mix%) x budgeted CM/unit. Favorable when a higher proportion of higher-margin products is sold than budgeted. SALES QUANTITY VARIANCE measures the effect of total units differing from budget at the constant budgeted mix and budgeted average CM: (Actual total units - Budgeted total units) x Budgeted mix% per product x budgeted CM/unit, summed; equivalently (Actual total units - Budgeted total units) x Budgeted weighted-average CM per unit. Relationship: Sales Mix Variance + Sales Quantity Variance = Sales Volume Variance. Worked example (accounting-simplified, Aliengear): budget 1,600 Turbox + 2,400 Speedo (40%/60% mix); actual 1,300 + 3,700 (total 5,000). Expected-at-actual-total-in-budget-mix = 2,000 Turbox + 3,000 Speedo. Turbox: (1,300-2,000) x CM 1,000 = -700,000 U; Speedo: (3,700-3,000) x CM 250 = +175,000 F; Sales Mix Variance = -525,000 Unfavorable (sold a worse mix — too many low-margin Speedos).
  - `spec:` Mix Var (per product) = (Actual units - Actual total units x Budgeted mix%) x Budgeted CM/unit. Quantity Var (per product) = (Actual total units - Budgeted total units) x Budgeted mix% x Budgeted CM/unit. Mix + Quantity = Sales Volume Variance
- **Direct Material Price Variance and Material Quantity (Usage) Variance** — MATERIAL PRICE VARIANCE = (Actual price per unit - Standard price per unit) x Actual quantity PURCHASED = (AP - SP) x AQP. Uses quantity PURCHASED (not used) so the variance is recognized when materials are acquired. Positive = unfavorable (paid more than standard). Worked (Saylor, Jerry's Ice Cream): (1.20 - 1.00) x 440,000 lbs = +88,000 Unfavorable. MATERIAL QUANTITY (USAGE) VARIANCE = (Actual quantity USED - Standard quantity allowed for actual output) x Standard price = (AQU - SQ) x SP. Uses standard price to strip out price effects. Positive = unfavorable (used more material). Worked: SQ = 2 lbs/unit x 210,000 units = 420,000 lbs; (399,000 - 420,000) x 1.00 = -21,000 Favorable. Terms: AP=actual price/unit, SP=standard price/unit, AQP=actual qty purchased, AQU=actual qty used, SQ=standard qty allowed = std qty per unit x actual units produced. Total direct materials variance = Price + Quantity (when AQP=AQU).
  - `spec:` Material Price Var = (AP - SP) x AQ purchased; Material Quantity/Usage Var = (AQ used - SQ allowed) x SP; SQ = std qty per unit x actual units produced
- **Direct Labor Rate Variance and Labor Efficiency Variance** — LABOR RATE VARIANCE = (Actual rate - Standard rate) x Actual hours = (AR - SR) x AH. Positive = unfavorable (paid a higher wage than standard). Worked (Saylor, Jerry's): (15 - 13) x 18,900 = +37,800 Unfavorable. LABOR EFFICIENCY VARIANCE = (Actual hours - Standard hours allowed) x Standard rate = (AH - SH) x SR. Uses standard rate to isolate the hours/productivity effect. Positive = unfavorable (used more hours). Worked: SH = 0.10 hr/unit x 210,000 = 21,000; (18,900 - 21,000) x 13 = -27,300 Favorable. Terms: AR=actual hourly wage, SR=standard hourly wage, AH=actual direct-labor hours worked, SH=standard hours allowed = std hours per unit x actual units produced. Total direct labor variance = Rate + Efficiency.
  - `spec:` Labor Rate Var = (AR - SR) x AH; Labor Efficiency Var = (AH - SH) x SR; SH = std hours per unit x actual units produced
- **Variable Overhead Spending Variance and Efficiency Variance** — These mirror the labor variances but use the variable-overhead rate per hour. VARIABLE OVERHEAD SPENDING VARIANCE = (Actual VOH rate per hour - Standard VOH rate per hour) x Actual hours = (AR - SR) x AH. Equivalent: Actual VOH cost - (AH x SR). Positive = unfavorable. VARIABLE OVERHEAD EFFICIENCY VARIANCE = (Actual hours - Standard hours allowed) x Standard VOH rate = (AH - SH) x SR. Driven entirely by the same activity-base efficiency (e.g. labor hours) as the labor efficiency variance. Positive = unfavorable. Terms: AR=actual VOH per activity hour, SR=standard/budgeted VOH per activity hour, AH=actual activity hours, SH=standard hours allowed for actual output. Total VOH variance = Spending + Efficiency.
  - `spec:` VOH Spending Var = (AR - SR) x AH = Actual VOH - (AH x SR); VOH Efficiency Var = (AH - SH) x SR
- **Fixed Overhead Budget (Spending) Variance and Fixed Overhead Volume Variance** — FIXED OVERHEAD SPENDING/BUDGET VARIANCE = Actual fixed overhead - Budgeted fixed overhead. There is NO efficiency variance for fixed overhead (fixed costs do not change with activity). Positive = unfavorable. Worked (Saylor, Jerry's): 136,000 - 140,280 = -4,280 Favorable. FIXED OVERHEAD VOLUME (PRODUCTION-VOLUME) VARIANCE = Budgeted fixed overhead - Applied fixed overhead, where Applied FOH = Standard fixed OH rate per unit x Actual units produced, and Std FOH rate per unit = Budgeted FOH / Budgeted (denominator) units. Equivalent: Std FOH rate per unit x (Budgeted units - Actual units produced). This is purely an artifact of absorption costing (over/under-applied capacity) and has no cash/spending meaning — it measures capacity utilization vs the denominator level. When actual production > budgeted (denominator), overhead is over-absorbed => Favorable; actual < budgeted => under-absorbed => Unfavorable. Worked (Saylor): 140,280 - 147,000 = -6,720 Favorable; equivalently 0.70/unit x (200,400 budgeted - 210,000 actual) = -6,720 Favorable. Total FOH variance = Spending + Volume.
  - `spec:` Fixed OH Spending/Budget Var = Actual FOH - Budgeted FOH. Fixed OH Volume Var = Budgeted FOH - Applied FOH = Std FOH rate/unit x (Budgeted units - Actual units); Std FOH rate/unit = Budgeted FOH / Budgeted units
- **Presentation: variance bridge / waterfall (Price-Volume-Mix), thresholds, and drill-down** — VARIANCE BRIDGE / WATERFALL: start from the Budget (or prior period) total bar, add signed contribution bars for each driver (typical order: Volume, then Mix, then Price, then FX, then individual cost variances), ending at the Actual total bar; bars must sum exactly to the total change. For a REVENUE PVM bridge over two periods the standard single-product decomposition is: Price effect = (Final price - Initial price) x Initial quantity; Volume effect = Initial price x (Final qty - Initial qty); with a residual Price x Volume interaction term that must be assigned (commonly added to price, or split). For MULTIPRODUCT, add a Mix effect; because sequential decomposition is ORDER-DEPENDENT (PVM vs VPM give different splits), rigorous tools average across calculation orders so the components are consistent and reconcile exactly to total revenue change — a key pitfall is double-counting mix within volume. Using contribution margin or gross profit instead of revenue in the bridge makes it tie to operating income. THRESHOLDS FOR FLAGGING: flag a line as material when BOTH an absolute dollar floor AND a percentage threshold are breached (common practice: |% variance| > 5-10% AND |$ variance| above a line-specific floor); use the dual test to avoid flagging large-% swings on tiny accounts and large-$ swings on huge stable accounts. DRILL-DOWN: store variances at the most granular grain (SKU/product x entity x cost center x account x period) and aggregate up dimension hierarchies; allow drill-down by product, region/entity, department/cost center, and account, with each driver attributable to an owner.
  - `spec:` PVM (single product): Price=(P1-P0)xQ0; Volume=P0x(Q1-Q0); interaction (P1-P0)x(Q1-Q0) assigned by convention. Multiproduct add Mix; average across decomposition orders to reconcile. Flag if |%var|>threshold(5-10%) AND |$var|>floor

**Recommendations**

- Store every variance as a SIGNED number computed consistently as (Actual - Standard/Budget), and derive the favorable/unfavorable flag from the account TYPE (cost vs revenue/CM), never from the sign alone. For cost accounts is_favorable = (variance < 0); for revenue/contribution/profit accounts is_favorable = (variance > 0). This single rule eliminates the most common sign bug.
- Implement the three-tier flexible-budget framework as first-class entities: Static (master) budget, Flexible budget (variable lines flexed to actual units at budgeted rates, fixed lines unchanged), and Actual. Then compute Sales Volume Variance = Flex - Static and Flexible Budget Variance = Actual - Flex, and assert Static Variance = sum of the two as an automated reconciliation/unit test.
- For cost variances, use the canonical price/quantity split: Price/Rate variance = (Actual rate - Std rate) x Actual quantity/hours; Quantity/Efficiency variance = (Actual qty/hours - Std qty/hours allowed) x Std rate. Always compute Standard-quantity-allowed = standard per unit x ACTUAL output (not budgeted output) — getting this base wrong is a frequent error.
- Make the multiplier basis for Sales Volume / Mix / Quantity variances configurable (contribution margin per unit recommended as default; allow profit-per-unit for absorption costing and price-per-unit for revenue-only views). Document the chosen basis on every report so price+volume(+mix) reconcile to the chosen total (revenue vs operating income).
- Treat Fixed Overhead Volume Variance distinctly: it is an absorption-costing capacity artifact (Budgeted FOH - Applied FOH), not a spending event. Label it clearly in the UI and exclude it from cash/spend-control dashboards to avoid misinterpretation; keep Fixed OH spending variance for actual-vs-budget spend control.
- Build the variance bridge/waterfall as the headline visualization (Budget -> Volume -> Mix -> Price -> cost drivers -> Actual) with bars that provably sum to the total change. For multi-period PVM, average the decomposition across calculation orders (or document a fixed convention for the price x volume interaction term) so components reconcile exactly and mix is not double-counted in volume.
- Flag materiality with a DUAL threshold: |% variance| above a configurable percent (default 5-10%) AND |$ variance| above a configurable dollar floor (optionally per-account). Guard the % calculation against budget = 0 (return N/A and rely on the dollar floor).
- Persist variances at the finest grain (period x entity x cost center x product/SKU x account x driver) so the same stored facts roll up for drill-down by any dimension; attach a driver/owner tag to each variance row to support accountable variance review.

**Open questions**

- Which costing basis will the product standardize on for sales-volume/mix variances by default — contribution margin (management-accounting standard) vs gross profit vs revenue — since this changes how the bridge reconciles (to operating income vs to revenue)?
- Will the tool support absorption costing (and therefore need the Fixed Overhead Volume Variance and a defined denominator/normal-capacity level), or operate purely on contribution-margin/variable costing where the FOH volume variance does not arise?
- Should the material price variance be recognized at PURCHASE (quantity purchased, isolates purchasing function earlier) or at USAGE (quantity used, so price and usage use the same quantity and reconcile cleanly to total material cost)? This is a deliberate accounting-policy choice with different journal-entry timing.
- What exact convention will be used to allocate the price x volume interaction (residual) term in multi-period PVM bridges — assign to price, assign to volume, split 50/50, or average across all decomposition orders — and should mix be computed only for multiproduct lines?
- What are the concrete default thresholds (percentage and dollar floors) per account category, and should thresholds be absolute dollar, percentage, statistical (e.g. control limits / standard deviations of historical variance), or a hybrid?
- For multiproduct mix/quantity variances, will products be grouped into comparable families (mix is only meaningful within substitutable product sets), and how will new/discontinued products with no budget baseline be handled in the mix calculation?

### existing-tools
The open-source finance landscape splits into three clusters, none of which combines all the capabilities our tool targets (autonomous statistical forecasting + the 4 corporate budgeting methods + budget-vs-actual variance analysis in one local tool).

(1) Personal-finance budgeting apps -- Actual Budget (MIT, local-first SQLite, envelope/zero-based budgeting, excellent import UX, NO statistical forecasting/variance), Firefly III (AGPLv3, PHP/Laravel + DB, double-entry, budgets/bills/rules, strong REST API + Data Importer, NO statistical forecasting -- its old prediction engine was removed), and Maybe Finance (AGPLv3, Rails, net-worth focus, ARCHIVED July 2025; community fork "Sure" under we-promise continues it).

(2) Plain-text accounting -- GnuCash (GPLv2+, double-entry, XML/SQLite, budget report = budgeted vs actual variance, no statistical forecasting) and Ledger/hledger (GPLv3, plain-text journal). hledger is the closest conceptual match: its `--budget` flag produces true budget-vs-actual variance reports and `--forecast` projects future balances, but BOTH are deterministic rule-based projections from user-authored periodic transactions -- there is NO statistical/time-series model.

(3) Billing & data platforms (adjacent, not budgeting) -- Lago (AGPLv3, usage-based billing/metering, not FP&A) and OpenBB (AGPLv3, financial data/quant platform, market data not internal budgeting).

Corporate FP&A: there is essentially NO mature, popular open-source corporate FP&A tool. The GitHub budgeting-and-forecasting topic is nearly empty (a single 0-star FinOps repo using Prophet). Statistical forecasting in OSS lives in general-purpose libraries we should embed rather than rebuild: statsmodels (BSD-3, ARIMA/SARIMAX/Holt-Winters), Prophet (MIT, additive trend+seasonality+holidays), and Nixtla statsforecast (Apache-2.0, fast AutoARIMA/ETS). This confirms a genuine gap: a single LOCAL tool that runs autonomous statistical forecasts AND supports the four formal budgeting methods AND does variance analysis does not exist today.

BORROW: double-entry integrity, rules engine, decoupled importer service, comprehensive API + webhooks. AVOID: AGPLv3 copyleft (network-use obligations) if we want permissive distribution; heavy self-host stack.
  - `spec:` License: AGPL-3.0. Stack: PHP 8 / Laravel + SQL DB. Import: CSV/CAMT.053, Spectre, Nordigen/GoCardless via separate Data Importer; REST JSON API + webhooks.
- **hledger / Ledger -- plain-text accounting; the ONLY tool here with both variance (--budget) and projection (--forecast), but rule-based not statistical** — Plain-text accounting: transactions live in a human-readable journal file the user controls; CLI + TUI (hledger-ui) + web (hledger-web). DATA MODEL: hierarchical accounts, multi-commodity/multi-currency, every posting balances (double-entry). KEY for our forecasting/variance dimension: hledger implements budget GOALS and forecasts via 'periodic transaction rules' -- an entry whose date is replaced by '~' plus a period expression (e.g. '~ monthly from 2013/01'). With '--budget', balance/report commands print budgeted-vs-actual side-by-side with variance shown as e.g. '396.46 USD [79% of 500 USD]' (true variance analysis). With '--forecast', those same rules generate temporary future transactions (not saved to the journal) that appear in all reports as if entered on their scheduled dates; '--cumulative' projects running balances. CRITICAL LIMITATION = our differentiator: both budget and forecast are DETERMINISTIC and user-authored -- the user must hand-write the recurring amounts; there is NO statistical/time-series model that learns amounts from history. BORROW: the elegant unified rule that doubles as budget goal AND forecast seed; transparent variance formatting; data-as-text/diffable storage. AVOID: requiring users to manually specify every projected figure; CLI-only ergonomics for non-technical users.
  - `spec:` License: GPLv3+. Budget rule: '~ monthly from 2013/01 / Expenses:Food 500 USD'. Reports: 'hledger balance --budget' (variance), 'hledger balance -M --forecast --cumulative' (projection). Forecast txns are not persisted.
- **GnuCash -- mature double-entry accounting with a budget-vs-actual report but no statistical forecasting** — Desktop double-entry accounting (personal + small business). DATA MODEL: default XML file; since 2.4 optional SQL backend (SQLite3/MySQL/PostgreSQL, still flagged experimental with edge-case data-loss reports). Every transaction debits one account and credits others by equal amounts. BUDGET/VARIANCE: GnuCash has a Budget feature and a Budget Report that shows budgeted and actual amounts in adjacent columns for each period (i.e. variance by period), plus Budgeted Income Statement and Budgeted Balance Sheet reports for small-business planning. FORECASTING: budgets are manually entered targets -- there is NO statistical forecasting engine. Extra features: multi-currency, customer/vendor tracking, invoicing, investment portfolios. BORROW: the budgeted vs actual side-by-side period report layout; budgeted financial statements (income statement / balance sheet) as an FP&A output. AVOID: XML-first storage and the immature SQL backend; desktop-only, dated UX.
  - `spec:` License: GNU GPL (GPLv2+). Storage: XML default, optional SQLite3/MySQL/PostgreSQL. Budget Report = budgeted vs actual adjacent columns per period.
- **Maybe Finance (ARCHIVED) + 'Sure' community fork -- net-worth + budgeting, cautionary tale** — Maybe was a venture-backed personal-finance app, open-sourced then ARCHIVED July 2025 (maybe-finance/maybe no longer maintained). Tech: Ruby on Rails web app + Sidekiq worker + PostgreSQL + Redis. FEATURES: aggregates bank/investment/real-estate/assets into net-worth view, transaction history, budgeting by category, goals, multi-currency with auto FX, optional OpenAI-powered insights/auto-categorization, Mint-export and CSV import. NO statistical forecasting or formal variance analysis. The community fork we-promise/sure continues development under AGPLv3 (PikaPods and others now recommend Sure over the archived original). LESSON for us: relying on a hosted/VC model is fragile; a genuinely local, self-contained tool with no mandatory cloud avoids this failure mode. BORROW: net-worth roll-up across asset classes; AI-assisted categorization as optional. AVOID: cloud/aggregator dependency; heavy 4-service deployment; AGPLv3 if permissive distribution is desired.
  - `spec:` License: AGPL-3.0. Stack: Rails + Sidekiq + PostgreSQL + Redis. Status: original archived 2025-07; active fork = github.com/we-promise/sure. Import: Mint export + CSV.
- **Lago and OpenBB -- adjacent, NOT budgeting/FP&A (scope boundary)** — Lago: open-source usage-based/subscription BILLING and metering API (AGPLv3, Ruby/Rails + Postgres, event ingestion, meters, aggregation per period/customer, invoicing, payment-gateway agnostic). It models metered usage and revenue, NOT internal budgets/forecasts -- relevant only if our tool ingests revenue/billing data as a forecasting input. OpenBB: open financial-DATA platform for analysts/quants/AI agents (license CHANGED to AGPL-3.0-only; ~100 data sources incl. FRED/IMF/BLS; Python + REST + Excel + MCP). It supplies market/macro data and analytics but is not an internal budgeting/variance tool. Neither competes directly; both are potential data-source integrations. BORROW from Lago: event/period aggregation model and 'connect-once consume-everywhere' API surface. BORROW from OpenBB: macro/market data connectors (FRED etc.) to enrich forecasts.
  - `spec:` Lago license: AGPL-3.0 (Rails). OpenBB license: AGPL-3.0-only (Python; ~100 sources incl. FRED/IMF/BLS).
- **Statistical forecasting libraries to EMBED (don't reinvent) -- this is where autonomous forecasting lives in OSS** — No finance app above does autonomous statistical forecasting; that capability comes from general-purpose Python time-series libraries we should depend on. statsmodels (License: Modified BSD-3, current v0.14.6 Dec 2025; provides ARIMA, Seasonal ARIMA/SARIMAX, ARIMAX, VARMAX, Exponential Smoothing / Holt-Winters, state-space models) -- permissive, ideal default engine for monthly financial series. Prophet by Meta (License: MIT; additive model = trend + yearly/weekly/daily seasonality + user-defined holiday/event effects, fit in Stan; great for seasonal series, handles missing data/outliers, easy autotuning) -- good 'autonomous' default for non-experts. Nixtla statsforecast (License: Apache-2.0; fast AutoARIMA/AutoETS, benchmarked far faster than pmdarima/Prophet/statsmodels) -- best for auto model selection at speed. RECOMMENDATION: ship statsmodels (BSD) + optionally statsforecast (Apache) as the core engine for license cleanliness; offer Prophet (MIT) as an optional seasonal model. All three are OSI-permissive and safe to bundle in a locally distributed tool.
  - `spec:` statsmodels: BSD-3-Clause, v0.14.6 (2025-12-05), ARIMA/SARIMAX/Holt-Winters. Prophet: MIT, additive trend+seasonality+holidays (Stan). Nixtla statsforecast: Apache-2.0, AutoARIMA/AutoETS.
- **The four budgeting methods (CFI taxonomy) -- our multi-method differentiator; no OSS tool implements all four** — The standard FP&A taxonomy (Corporate Finance Institute) is four methods: (1) Incremental -- take last period's actuals and add/subtract a percentage (simplest, most common; maps naturally to our statistical baseline + growth-rate adjustment). (2) Activity-Based -- set a target output then derive the activities and their costs required to hit it. (3) Value-Proposition -- allocate spend by the value/ROI each item delivers (justify why each cost adds value). (4) Zero-Based -- justify every expense from a zero base each period (no automatic carryover). Existing OSS budgeting tools implement at most ONE informal flavor: Actual = envelope (a zero-based variant); Firefly/GnuCash/hledger = simple period limits/incremental-style targets. NONE offers all four as selectable methods, and none pairs them with statistical forecasting or variance analysis. This is the core product gap our tool fills. Rolling forecasts (continuously updated projections rather than a static annual budget) are a related FP&A best practice worth supporting on top of the statistical engine.
  - `spec:` Methods: Incremental (prior actuals +/- %), Activity-Based (target->activities->cost), Value-Proposition (allocate by ROI/value), Zero-Based (justify from 0 each period). Plus rolling forecast as an ongoing-update mode.
- **Corporate FP&A open-source landscape is nearly empty -- confirms the gap** — Searching GitHub's budgeting-and-forecasting / financial-forecasting / financial-planning topics and the OpenAlternative FP&A category surfaces NO mature, popular, purpose-built open-source corporate FP&A application. The budgeting-and-forecasting topic effectively contains one 0-star hobby repo (a FinOps cloud-cost forecaster using Prophet for variance analysis). The FP&A category on OpenAlternative lists only OpenBB (data/quant platform, not budgeting) and Bigcapital (cloud accounting/bookkeeping, GitHub ~3.7k stars -- accounting + reporting + multi-currency, not statistical forecasting or multi-method budgeting). CONCLUSION: combining (a) autonomous statistical forecasting, (b) all four formal budgeting methods, and (c) budget-vs-actual variance analysis in a single LOCAL tool is genuinely unoccupied territory in open source. Closest partial overlaps are hledger (variance + deterministic projection) and Actual (one budgeting flavor + great import UX).
  - `spec:` No mature OSS corporate FP&A app exists. Closest: hledger (variance + rule-based forecast), Actual (envelope + imports). Bigcapital = accounting, not FP&A.

**Recommendations**

- Embed proven OSS forecasting engines rather than building from scratch: default to statsmodels (BSD-3: ARIMA/SARIMAX/Holt-Winters) for license cleanliness, add Nixtla statsforecast (Apache-2.0: AutoARIMA/AutoETS) for fast autonomous model selection, and offer Prophet (MIT: trend+seasonality+holidays) as an optional model for strongly seasonal series. All are OSI-permissive and safe to bundle in a locally distributed tool.
- Adopt Actual Budget's storage and import patterns: a single local SQLite file per workspace, full undo/redo, and a rich multi-format importer (CSV, OFX/QFX, QIF, CAMT.053) plus app-specific migrators -- import UX is where these tools win adoption and is largely orthogonal to our forecasting differentiator.
- Borrow hledger's unified 'budget rule' concept that simultaneously serves as a budget goal and a forecast seed, and copy its transparent variance presentation (actual vs budgeted with absolute and % consumed), but REPLACE its hand-authored deterministic amounts with statistically forecasted baselines the user can override -- this is the precise wedge between hledger and our tool.
- Implement all four CFI budgeting methods as explicitly selectable modes (incremental, zero-based, activity-based, value-proposition) plus a rolling-forecast mode; map the statistical forecast to seed the incremental/rolling baseline, since no existing OSS tool offers method choice. This multi-method + statistical pairing is the core differentiation to message in product positioning.
- Reuse Firefly III's data-model ideas where they add rigor (double-entry integrity, a rules/automation engine, a decoupled importer, comprehensive API+webhooks) but keep the app fully local with no mandatory cloud -- Maybe Finance's archival shows cloud/VC dependence is a survival risk for finance tools.
- Choose a permissive license (MIT/BSD/Apache) for our distributable tool. The strongest finance OSS peers (Firefly III, Maybe/Sure, Lago, OpenBB) are AGPL-3.0, which imposes network-copyleft; staying permissive (like Actual) eases adoption and embedding. Verify the bundled forecasting libs (all permissive) for compatibility -- no GPL/AGPL forecasting dependency is required.
- Offer FP&A-grade outputs GnuCash hints at but none combine with forecasting: budgeted income statement and balance sheet, period-by-period budgeted-vs-actual variance with drill-down, and forecast confidence intervals (a feature absent from every tool reviewed -- statistical models give us prediction intervals for free).
- Position macro/market data (OpenBB connectors: FRED/IMF/BLS) and billing/usage data (Lago-style event aggregation) as optional forecast inputs/integrations rather than rebuilding them, to enrich forecasts without scope creep.

**Open questions**

- What exact variance methodology should we surface (simple actual-minus-budget, % of budget like hledger, or full price/volume/mix variance decomposition used in corporate FP&A)? CFI/standard cost-accounting variance formulas should be confirmed in a separate dimension.
- Is the target user household/SMB or enterprise FP&A? This determines whether double-entry rigor (Firefly/GnuCash) is required or whether a simpler categorized-cashflow model suffices, and how heavyweight the four budgeting methods need to be.
- Do we need true double-entry bookkeeping under the hood, or is a single-entry categorized ledger enough for budgeting + forecasting + variance? Firefly/GnuCash/hledger all chose double-entry; Actual did not strictly require it for envelope budgeting.
- Which import formats are must-have for our audience (bank CSV is universal; OFX/QFX/QIF for legacy; CAMT.053 for European/corporate bank statements)? Need to confirm whether corporate users expect ERP/GL exports (e.g., from QuickBooks, Xero, SAP) rather than personal-bank formats.
- Should forecasting run fully autonomously (auto-select model + hyperparameters via statsforecast AutoARIMA) or expose model choice? Need to validate accuracy/UX tradeoff for non-technical finance users on typical short monthly financial series (often <60 points, which can challenge ARIMA/Prophet).
- Confirm Prophet's current maintenance status and Python/Stan dependency footprint (cmdstanpy) before bundling -- its heavier install may argue for statsmodels/statsforecast as the only hard dependency with Prophet optional.

### data-model-template
A complete, build-ready data model and upload template for a budgeting & forecasting tool, grounded in standard accounting (chart of accounts, normal-balance sign conventions) and Kimball-style dimensional modeling (star schema, conformed dimensions, atomic monthly grain). RECOMMENDED ARCHITECTURE: Store facts in a single LONG/tidy "entries" fact table in SQLite (one row per account x period x dimension-combo x scenario x version), keyed by smart integer period_key (YYYYMM); store all monetary amounts as INTEGER minor units (cents) to avoid float error, with a currency + a precomputed REAL amount only for display. Use a WIDE upload template (accounts down rows, periods across columns) for human-friendly Excel/CSV ingestion, with a parallel LONG template for system/ETL feeds; the importer pivots wide->long on load. Scenarios (Actual/Budget/Forecast) are an explicit dimension; budgets carry a budget_version_id and forecasts a forecast_run_id so multiple iterations coexist. Variance results are computed (Actual - Budget) with sign flipped for expense-type accounts so "favorable" is always positive. The package below gives: COA + hierarchy with numbering ranges and normal balances; dimension catalog; a date/period dimension with fiscal-year handling and period keys; long-vs-wide rationale; exact CSV column lists, types, example rows, and validation rules for actuals vs budget/forecast; a manual-entry grid model; and the full SQLite DDL (accounts, dimensions, periods, entries, budget_versions, forecast_runs, variance_results).

**Key findings**

- **Chart of accounts: types, numbering ranges, and category hierarchy** — A chart of accounts (COA) lists all financial accounts grouped into the five primary types: Assets, Liabilities, Equity, Revenue, and Expenses (with Gains/Losses as a variable subset). Accounts are arranged in financial-statement order: balance-sheet accounts (assets, liabilities, equity) first, then income-statement accounts (revenue, expenses). Standard block numbering: 1000s=Assets, 2000s=Liabilities, 3000s=Equity, 4000s=Revenue, 5000s=Expenses/COGS (direct costs of producing goods/services), 6000s=Operating Expenses (OpEx, costs to run the business not directly tied to delivery), 7000s=Other income/expense, intercompany & taxes. For an FP&A tool I model a 3-level hierarchy on the accounts table: statement_section (P&L vs Balance Sheet) -> account_type (REVENUE, COGS, OPEX, OTHER_INCOME, OTHER_EXPENSE, ASSET, LIABILITY, EQUITY) -> account_category (e.g. 'SaaS Revenue', 'Cloud Hosting', 'Salaries & Wages') -> account (leaf, the postable line). Each account stores parent_account_id for arbitrary roll-up and a is_postable flag (only leaves accept entries; roll-up nodes are computed).
  - `spec:` account_type enum: REVENUE | COGS | OPEX | OTHER_INCOME | OTHER_EXPENSE | ASSET | LIABILITY | EQUITY. Numbering: Assets 1000-1999, Liabilities 2000-2999, Equity 3000-3999, Revenue 4000-4999, COGS 5000-5999, OpEx 6000-6999, Other/Tax 7000-7999.
- **Sign conventions (normal balance) and how amounts are stored in the fact table** — Normal balances: Assets and Expenses (incl. COGS, OpEx) are DEBIT-normal (increase with debit); Liabilities, Equity, and Revenue are CREDIT-normal (increase with credit). For an FP&A/reporting tool you do NOT store debit/credit; instead store a single signed amount per row using a consistent convention, then store normal_balance and a sign_factor on the account so reports and variance can normalize. RECOMMENDED CONVENTION: store amounts as 'natural reporting sign' = positive for revenue and positive for expenses (i.e. a P&L line shows revenue 100,000 and salaries 40,000 as positive magnitudes), and let the report layer subtract expenses from revenue. Persist normal_balance ('DEBIT'|'CREDIT') and a derived sign_factor (+1 for revenue/income, -1 for expense/COGS/opex) so net income = SUM(amount * sign_factor). This keeps uploads intuitive (users type positive numbers) while the engine computes signed math correctly.
  - `spec:` Normal balance map: ASSET/EXPENSE/COGS/OPEX/OTHER_EXPENSE = DEBIT; LIABILITY/EQUITY/REVENUE/OTHER_INCOME = CREDIT. P&L sign_factor: REVENUE/OTHER_INCOME=+1; COGS/OPEX/OTHER_EXPENSE=-1. Net Income = SUM(amount_minor * sign_factor).
- **Dimensions: department/cost center/project/region + entity, plus scenario & versioning** — Model business dimensions as conformed dimension tables shared by all facts (Kimball): department (cost center), project, region, and legal entity. Each is a small, slowly-changing reference table with a surrogate integer PK, a stable business code, a name, and optional parent for hierarchy/roll-up. SCENARIO is an explicit dimension with values ACTUAL, BUDGET, FORECAST (extensible: PRIOR_YEAR, PLAN, WHAT_IF). Each fact row carries department_id, project_id, region_id, entity_id, scenario, plus version pointers. VERSIONING: budgets get a budget_version_id FK to budget_versions (e.g. 'FY26 Original', 'FY26 Q2 Reforecast'), and forecasts get a forecast_run_id FK to forecast_runs (timestamped run with method + assumptions). Actuals have NULL version pointers (there is only one truth). This lets many budget/forecast iterations coexist and be compared without overwriting. Use a 'current/active' flag on budget_versions so reports default to the approved version. Allow a sentinel dimension member (e.g. department code 'UNALLOC', id=0) for unallocated/company-wide amounts so every fact row has non-null FKs.
  - `spec:` scenario enum: ACTUAL | BUDGET | FORECAST. Fact FKs: account_id, period_key, entity_id, department_id, project_id, region_id, scenario, budget_version_id (nullable), forecast_run_id (nullable), currency. Sentinel members id=0 = 'Unallocated/All'.
- **Time: period granularity, fiscal year, and smart period keys** — Best practice is to model at the most granular planning period (MONTHLY) and roll up to quarter/year, because monthly facts aggregate cleanly to any higher grain but the reverse requires allocation. Build a periods (date/calendar) dimension at MONTH grain with a smart integer key. Use period_key INTEGER as YYYYMM (e.g. 202604 for Apr 2026) for the month grain - it is human-readable, sorts correctly, joins fast, and is the de-facto Kimball 'smart date key' pattern (analogous to the YYYYMMDD day key). Store calendar attributes (year, month_num, month_name, quarter, period_start_date, period_end_date in ISO 8601) AND fiscal attributes (fiscal_year, fiscal_quarter, fiscal_period_num 1-12) computed from a configurable fiscal_year_start_month. Roll-ups are GROUP BY on fiscal_year / fiscal_quarter. For a company whose FY starts in April, April 2026 = fiscal_year 2027 (or 2026 depending on convention - make it a setting), fiscal_period_num=1. Keep period_key as the only time FK on facts; never store free-text dates on facts.
  - `spec:` period_key = year*100 + month (YYYYMM, INTEGER). fiscal_period_num = ((month_num - fy_start_month + 12) % 12) + 1. fiscal_quarter = CEIL(fiscal_period_num/3). fiscal_year (April-start, label-by-end convention) = year + (month_num >= fy_start_month ? 1 : 0). period_start/end stored as ISO 8601 'YYYY-MM-DD'.
- **Storage layout: LONG (tidy) for the DB, WIDE for the upload template** — Two distinct concerns. (1) DATABASE: use LONG/tidy layout (Hadley Wickham: each variable is a column, each observation a row). One fact row = one account x one period x one dimension-combo x one scenario/version, with a single 'amount' value column and a 'period_key' key column. Rationale: adding a new month is INSERTs not schema changes (a wide DB needs a new column per month - DDL churn, brittle, hits column limits); long form makes GROUP BY/aggregation, filtering, variance joins, and indexing trivial; it is the natural shape for SQL analytics. (2) UPLOAD TEMPLATE: use WIDE layout (accounts down the rows, Jan..Dec across columns) because that is how finance teams build budgets in Excel and read P&Ls; it minimizes row count and data entry friction. The importer PIVOTS wide->long on ingest (each period column becomes N long rows). Also offer a LONG csv template for programmatic/system feeds (ERP exports) since those are already row-per-transaction. NOTE the tidy caveat: for a single fixed annual view wide can be 'tidy' too, but for a multi-period planning DB long is unambiguously correct.
  - `spec:` DB (long) columns: account_code, period_key, entity, department, project, region, scenario, version, amount. Upload (wide) columns: account_code, ...dimensions..., 2026-01, 2026-02, ... 2026-12. Pivot: UNPIVOT period columns -> (period_key, amount) rows.
- **Monetary value storage: INTEGER minor units (cents), never float** — IEEE-754 floats cannot represent most decimal money exactly (e.g. 2.78 stores as 2.7799999...), causing rounding drift in summed forecasts. Store amounts as INTEGER minor units (cents): $12.34 -> 1234. This gives exact addition/comparison and is compact (BIGINT/INTEGER). SQLite has no native decimal/money type (only NULL/INTEGER/REAL/TEXT/BLOB storage classes with type affinity), which makes the integer-cents approach especially appropriate there. Schema: amount_minor INTEGER NOT NULL (signed, in the account's reporting convention), currency TEXT (ISO 4217, e.g. 'USD'), and optionally minor_unit_scale INTEGER (default 2) for currencies with different decimal places. Convert to display decimals only at the presentation layer (amount_minor / 10^scale). Avoid REAL for stored amounts; REAL is acceptable only for derived percentages (variance_pct) where exactness is not required.
  - `spec:` amount_minor INTEGER = round(decimal_amount * 10^minor_unit_scale). Display = amount_minor / 10^minor_unit_scale. currency = ISO 4217 code, default 'USD', scale default 2.
- **ACTUALS upload template: exact columns, types, example rows** — WIDE template (default, Excel/CSV). Header row 1 = column names; period columns are ISO month labels. Columns: account_code (TEXT, required, must exist & be postable), account_name (TEXT, optional/informational only - ignored on import or used to warn on mismatch), entity_code (TEXT, required), department_code (TEXT, required - use 'UNALLOC' if none), project_code (TEXT, optional, default 'NONE'), region_code (TEXT, optional, default 'ALL'), currency (TEXT ISO 4217, optional, default workspace currency), then one numeric column per period named YYYY-MM (e.g. 2026-01 ... 2026-12). Numeric cells: plain numbers, no currency symbols, no thousands separators, '.' decimal only; blank = no entry (treated as NULL, not 0); negatives allowed for contra/credit memos. For ACTUALS scenario is implicitly ACTUAL and version columns are absent. LONG variant columns: account_code, entity_code, department_code, project_code, region_code, period (YYYY-MM or YYYY-MM-DD), amount, currency. Example WIDE rows below.
  - `spec:` WIDE example: 4000,SaaS Revenue,US01,SALES,PROJ-X,NA,USD,120000.00,125000.50,131000,... | 6000,Salaries,US01,ENG,NONE,NA,USD,80000,80000,82500,... | 5000,Cloud Hosting,US01,ENG,PROJ-X,NA,USD,15000,15250,16000,...  LONG example: 4000,US01,SALES,PROJ-X,NA,2026-01,120000.00,USD
- **Validation rules for uploads (apply at ingest, row+cell level, fail fast)** — Required-field checks: account_code, entity_code, department_code, and at least one period column must be present and non-empty. Account-must-exist: every account_code must match an existing accounts.account_code AND accounts.is_postable=1 (reject roll-up nodes). Dimension-must-exist: entity/department/project/region codes must resolve to active dimension members (auto-create optional but default reject with clear error). Date/period format: period columns/values must be ISO 8601 'YYYY-MM' (month) or 'YYYY-MM-DD' (coerced to month); reject ambiguous locale dates. Numeric parsing: strip nothing - reject values containing currency symbols, %, or thousands separators; require '.' as decimal; empty = NULL (not 0); enforce numeric type (CSV is typeless so explicit type-check prevents strings landing in numeric columns). Duplicate/period checks: reject duplicate (account, entity, dept, project, region, period, scenario, version) keys within a file; on load, UPSERT against the unique index (replace or sum per policy). Period-window check: warn/reject periods outside the open fiscal window or already locked/closed periods. Sign check: optionally warn when an expense account has a negative value or revenue is negative. Validate streaming row-by-row against the schema and return a per-row error report rather than failing the whole file silently.
  - `spec:` Hard reject conditions: missing required col; unknown/non-postable account_code; unknown dimension code; non-ISO date; non-numeric or formatted numeric in period cell; duplicate natural key in file; write to locked period. Unique key = (account_id, period_key, entity_id, department_id, project_id, region_id, scenario, budget_version_id, forecast_run_id).
- **How BUDGET and FORECAST uploads differ from ACTUALS** — Same WIDE/LONG shape and same validations, plus version metadata. BUDGET upload requires a budget_version (either a metadata header field 'budget_version=FY26 Original' or a dedicated column); the importer looks up/creates the budget_versions row and stamps budget_version_id on every fact, scenario='BUDGET'. Typically budgets cover full future fiscal year(s) and disallow writing to closed/actualized periods. FORECAST upload requires a forecast_run identity (run label, as-of date, method); importer creates a forecast_runs row and stamps forecast_run_id, scenario='FORECAST'. Forecasts often mix: actuals for elapsed months + forecast for remaining months ('actuals + forecast' / rolling forecast) - the tool stores only the forecast rows under FORECAST scenario; reporting overlays ACTUAL where present. Re-uploading the same version/run UPSERTs (overwrites) that version's cells, preserving prior versions/runs untouched. Actuals never carry a version. Provide one downloadable template per scenario, pre-seeded with the active COA rows and the correct period columns for the target fiscal year so users only fill numbers.
  - `spec:` Budget extra: budget_version (header or column) -> budget_versions(version_id, version_name, fiscal_year, status, is_active, created_by, created_at). Forecast extra: forecast_run (label + as_of_date + method) -> forecast_runs(run_id, run_label, as_of_period_key, method, created_at). Actuals: no version field.
- **Manual-entry data model (accounts x periods grid)** — The UI is a 2-D grid: rows = postable accounts (optionally grouped by category with computed subtotals), columns = periods (the target fiscal year's 12 months, plus a Total column = SUM, read-only). The user selects a context (entity, department, project, region, scenario, and version/run) before editing - that context fixes the dimension FKs so each editable cell maps to exactly ONE long fact row: cell(account, period) under the chosen context -> entries row. Editing a cell writes/UPSERTs one long row keyed by the full natural key; clearing a cell deletes the row (or sets NULL). Subtotal/roll-up rows are computed from leaf cells, never stored as facts. Optimistic concurrency via updated_at; an audit/changelog row per edit (who, when, old->new) is recommended. The grid is therefore a thin WIDE presentation over the LONG store - same pivot logic as the wide upload, applied per-cell. Support copy-from: seed a budget grid from prior-year actuals * growth, or copy a forecast run into a new run.
  - `spec:` Grid cell (account_id, period_key) + fixed context (entity_id, department_id, project_id, region_id, scenario, version_id/run_id) -> 1 entries row. Total column = SUM(amount_minor) over periods. Subtotals = SUM over child accounts (computed, not stored).
- **Variance results: definition, sign normalization, storage** — Variance = Actual - Budget (or Actual - Forecast). Absolute variance and % variance = (Actual - Budget)/ABS(Budget). The critical FP&A nuance: 'favorable' differs by account type. For REVENUE/income, Actual > Budget is FAVORABLE (positive). For EXPENSE/COGS/OPEX, Actual < Budget is FAVORABLE. To present a single intuitive number, store both the raw variance AND a 'favorable-normalized' variance = raw_variance * sign_factor (sign_factor +1 for revenue, -1 for expense), so a positive favorable_variance always means 'good'. Also store a variance_status enum (FAVORABLE/UNFAVORABLE/NEUTRAL). **Recommendations**

- DB layer: store facts in a single LONG/tidy 'entries' table (one row per account x period x dimension-combo x scenario x version). Adding periods becomes INSERTs, not DDL; aggregation, filtering, variance joins, and indexing are all trivial.
- Amounts: store as INTEGER minor units (cents) with a currency (ISO 4217) and minor_unit_scale; never use REAL/float for stored money. SQLite has no decimal type, so integer-cents is the exact, compact choice. Convert to decimals only for display.
- Time: model at MONTHLY grain with a periods dimension and a smart INTEGER period_key = YYYYMM; precompute fiscal_year/fiscal_quarter/fiscal_period_num from a configurable fiscal_year_start_month and roll up via GROUP BY.
- Scenario as an explicit dimension (ACTUAL/BUDGET/FORECAST); version budgets via budget_version_id -> budget_versions and forecasts via forecast_run_id -> forecast_runs so multiple iterations coexist and are comparable. Enforce the scenario/version consistency with a CHECK constraint.
- Sign handling: store positive natural-magnitude amounts (users type positives) and persist normal_balance + sign_factor on accounts; compute net income as SUM(amount*sign_factor) and favorable variance as variance*sign_factor so 'good' is always positive regardless of revenue vs expense.
- Upload template: ship a WIDE Excel/CSV (accounts down rows, YYYY-MM period columns across) pre-seeded with the active COA and target-year periods; pivot wide->long on import. Also provide a LONG CSV template for ERP/system feeds.
- Validation: enforce required fields, account-must-exist + is_postable, dimension-must-exist, ISO 8601 dates (YYYY-MM), plain numeric parsing (reject symbols/separators), blank=NULL not 0, in-file duplicate-key rejection, UPSERT on the natural-key unique index, and lock writes to closed periods. Validate row-by-row and return a per-row error report.
- Budget/forecast uploads reuse the same shape + validations but require version metadata (budget_version header/column; forecast run label + as-of date + method); re-upload UPSERTs only that version/run and leaves prior versions intact. Actuals carry no version.
- Manual entry is a WIDE grid (accounts x periods) over a fixed context (entity/department/project/region/scenario/version); each editable cell maps to exactly one long entries row via the natural key; subtotals/roll-ups are computed, never stored as facts; add an audit changelog and optimistic-concurrency updated_at.
- Variance: materialize a variance_results table keyed at the fact grain comparing two scenarios/versions; store variance_minor, favorable_variance_minor, variance_pct (guard divide-by-zero with NULLIF), and a FAVORABLE/UNFAVORABLE/NEUTRAL status; refresh after each load.
- Use a sentinel '0 = Unallocated/All' member in every dimension so all fact FKs are NOT NULL, keeping joins simple and avoiding NULL-handling branches in aggregation.
- Enforce the fact natural-key UNIQUE index (account, period, all dims, scenario, version) - it simultaneously prevents duplicates, powers idempotent UPSERT imports, and is the join target for the grid and variance engine.

**Open questions**

- Multi-currency policy: is this single-currency (store one currency, simplest) or multi-entity multi-currency? If the latter, decide whether to store amounts in transaction currency + FX rates table and a reporting-currency conversion (adds an fx_rates dimension and a functional-currency amount column).
- Fiscal-year labeling convention: when FY starts April, is FY labeled by its starting calendar year or ending year? This changes the fiscal_year formula and must be a configurable workspace setting.
- Re-upload semantics per cell: should re-importing a (version, key) cell REPLACE the value or SUM/accumulate? Replace is the safe default for planning grids; sum suits transaction-level actual feeds - may need to be configurable per upload.
- Account/dimension hierarchy depth: is a single parent_id self-reference (ragged hierarchy) sufficient, or are fixed levels (e.g. type>category>account) required for reporting? Self-reference is more flexible; fixed levels are easier to validate.
- Balance-sheet handling: this design optimizes the P&L (flows, sign_factor math). Balance-sheet accounts are stock/point-in-time (period-end balances, not summable across periods) - confirm whether the tool needs full BS + cash flow, which would warrant a separate periodic-snapshot treatment or a balance_type flag (FLOW vs BALANCE) on accounts.
- Slowly changing dimensions: if a department/cost-center gets reorganized mid-year, do we need historical attribute tracking (Kimball SCD Type 2 with effective dates) or is overwriting (Type 1) acceptable? Type 1 is simpler and usually fine for a budgeting tool.
- Allocations/drivers: will the tool need driver-based allocation (e.g. spread shared OpEx across departments by headcount)? If so, an allocation_rules table and an allocation step before variance would be needed - out of scope for the core schema but worth reserving.
- Granularity below month: any need for weekly/13-period (4-4-5) calendars or daily actuals? The smart period_key would change (e.g. add a separate day-grain periods table) if so.

### Focus: forecasting-methods
Overall the forecasting-methods research is accurate, well-sourced, and build-ready. All named Python APIs were verified against primary docs and behave as described: statsmodels SARIMAX (get_forecast/conf_int) and ExponentialSmoothing/ETSModel, pmdarima auto_arima (seasonal/m/stepwise/test=kpss/seasonal_test=ocsb|ch, predict return_conf_int), Nixtla statsforecast AutoARIMA/AutoETS/AutoTheta/AutoCES with season_length and conformal level=[...] intervals, and prophet Prophet (ds/y, interval_width=0.8, yhat/yhat_lower/yhat_upper). The statistical claims about what each model captures are correct, including the SARIMAX-with-exog 'regression with SARIMA errors' interpretation (confirmed by statsmodels' own docs). The methodology is correct: rolling-origin CV is the right time-series validation (no leakage; train only before each cutoff), and the error-metric formulas (MAE/RMSE/MAPE/sMAPE/MASE) are stated correctly with MASE rightly recommended as the scale-free primary metric.\n\nThree nuances the build team should note (none are errors that block the design): (1) SHORT-SERIES/SEASONALITY GATE — the '>=2 full seasonal cycles (>=24 monthly points)' rule is a reasonable engineering heuristic, but the cited Hyndman sources explicitly reject FIXED minimum-observation thresholds as 'misleading and unsubstantiated' and only assert the theoretical floor of 'more observations than parameters.' Implement the 2-cycle gate as a product heuristic, not as Hyndman-endorsed theory, and lean on the auto-models' own AICc/seasonal-difference tests plus STL Fs. (2) PROPHET INTERVALS — default yhat_lower/upper (MAP, mcmc_samples=0) capture trend uncertainty + observation noise but not full parameter uncertainty, so they can be optimistic; this reinforces the research's own recommendation to prefer conformal intervals validated on CV coverage. (3) CLASSIC HOLT-WINTERS — statsmodels' holtwinters.ExponentialSmoothing gives point forecasts but NO analytic prediction intervals; use ETSModel/state-space or conformal when intervals are required (the research states this correctly). Also worth flagging operationally: weekly data has non-integer seasonal period (~52.18) and multi-seasonal series (daily with weekly+yearly) may need MSTL/Fourier rather than a single integer m; and conformal calibration requires n_windows*h < series length, which interacts with the short-series fallback. A minor k-fold subtlety: random k-fold is theoretically valid specifically for pure autoregressive models (Bergmeir/Hyndman/Koo 2015), but rolling-origin remains the correct general default for heterogeneous FP&A series. No factual refutations were found.
